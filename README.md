Автоматизированное тестирование сайта с использованием Selenium
В данном проекте реализовано автоматизированное тестирование веб-приложения с помощью Selenium WebDriver. Тесты покрывают ключевой пользовательский сценарий: проверка доступности страниц, добавление товаров в корзину, корректность отображения цен и сравнение с акционными предложениями.

Структура тестового набора
1. Папка pages/
В этой директории реализованы Page Object Model (POM) — классы, представляющие различные страницы тестируемого сайта. Каждый класс инкапсулирует:

локаторы элементов (CSS, XPath)

методы взаимодействия с элементами (клик, ввод текста, получение значений)

проверки состояния страницы

Это позволяет централизованно управлять селекторами и упрощает поддержку тестов при изменении интерфейса.

2. Файл test_main_page.py
Выполняет базовую проверку доступности главной страницы:

открывает стартовую страницу сайта

проверяет, что основные элементы загружаются (заголовок, меню, блоки контента)

убеждается, что страница не возвращает ошибок HTTP (4xx/5xx) и доступна для дальнейшего тестирования

Этот тест служит дымовым (smoke test) — быстрая проверка, что с главной страницей всё в порядке.

3. Файл test_product_page.py
Основной файл с расширенными сценариями тестирования. Он проверяет полный цикл работы с товарами и корзиной. Включает следующие функции:

Добавление товара в корзину
Тест находит выбранный товар на странице каталога, нажимает кнопку «В корзину» и проверяет, что в корзине появился нужный товар.

Решение задачи с сайта
В ходе тестирования автоматически обрабатывается всплывающее задание (например, капча или математический пример), чтобы продолжить проверку основного сценария.

Сравнение цены в корзине с ценой на странице распродажи
Критически важная проверка:

Берется цена товара на странице акции/распродажи

Товар добавляется в корзину

Цена в корзине сравнивается с акционной ценой

Тест падает, если цены различаются (что может указывать на ошибку в логике сайта или в расчете скидки)

Принципы разработки кода
Код написан с соблюдением следующих принципов:

DRY (Don’t Repeat Yourself) – общие методы вынесены в базовые классы или утилиты, избегается дублирование логики.

SOLID – классы имеют одну зону ответственности, используется внедрение зависимостей (например, передача драйвера в конструктор страниц).

Page Object Model – отделение логики тестов от деталей реализации интерфейса.

Читаемость – названия тестов отражают их цель, тесты независимы и могут запускаться в любом порядке.

Запуск тестов

pytest test_main_page.py
pytest test_product_page.py
Требования
Python 3.7+

Selenium WebDriver
